## Key decisions

## What a future agent needs to know

### Beads architecture
- Issues live in a Dolt database inside `.beads/`. Dolt is a SQL database with git-like versioning.
- The `.beads/` directory has its own `.gitignore` that excludes runtime files but tracks config and metadata.
- On a new machine: `brew install beads`, then `bd info` auto-discovers the database after `git pull`.
- `bd sync` is referenced in AGENTS.md but doesn't exist as a standalone command -- sync happens through Dolt's own mechanism.

### Beads workflow
```
bd ready          # See actionable issues
bd children <id>  # See issue tree under an epic
bd show <id>      # Full issue details
bd update <id> --claim  # Start working on it
bd close <id> --reason "Fixed in commit abc123"
```

### Health check deduplication
The `/health-check` skill creates an epic per run (labeled `health,health-run`). Individual findings become children. Before creating new beads, it checks existing open beads for duplicates via title similarity.
